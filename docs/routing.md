# Routing Best Practices

> Enterprise routing design recommendations for MikroTik RouterOS VPN deployments using WireGuard, OpenVPN and L2TP/IPsec.

---

# Overview

Routing is one of the most critical components of every VPN deployment.

Even when authentication, encryption and firewall policies are correctly configured, incorrect routing may prevent VPN clients from reaching internal resources.

A properly designed routing strategy ensures:

- Predictable traffic flow
- Stable VPN connectivity
- Efficient network utilization
- Simplified troubleshooting
- Better security
- Easier scalability

This document explains the routing concepts commonly used in MikroTik RouterOS VPN environments.

---

# Understanding Routing

Routing determines how packets travel between networks.

Every VPN connection creates at least one additional network that must be reachable.

Example:

```text
Corporate LAN
192.168.110.0/24

↓

WireGuard VPN
192.168.120.0/24

↓

Remote User
```

The router must know how traffic should travel between these networks.

Incorrect routes typically result in:

- VPN connects successfully
- Internal resources remain unreachable
- DNS resolution failures
- One-way communication
- Intermittent connectivity

# Full Tunnel vs Split Tunnel

One of the first routing decisions is determining which traffic should pass through the VPN.

---

## Full Tunnel

All traffic generated by the VPN client is sent through the encrypted tunnel.

```text
Internet

↓

VPN Client

↓

MikroTik

↓

Corporate Firewall

↓

Internet
```

Advantages:

- Centralized security
- Consistent filtering
- Secure public Wi-Fi usage
- Better compliance

Disadvantages:

- Increased Internet bandwidth usage
- Higher latency
- Additional load on the VPN gateway

---

## Split Tunnel

Only traffic destined for internal resources traverses the VPN.

Internet traffic continues to use the client's local Internet connection.

```text
Internal LAN

↓

VPN Tunnel

Internet

↓

Local ISP
```

Advantages:

- Better performance
- Lower bandwidth consumption
- Reduced router load

Disadvantages:

- Reduced centralized security
- Internet traffic bypasses corporate controls

For most business environments, Split Tunnel is the recommended deployment model.

---

# Static Routing

Static routes define how packets reach remote networks.

They are suitable for:

- Small businesses
- Branch offices
- Stable environments
- Permanent VPN tunnels

Advantages include:

- Simple configuration
- Predictable behavior
- Minimal resource usage

Static routing is recommended unless dynamic routing is required.

---

# Dynamic Routing

Larger organizations may require dynamic routing protocols.

Examples include:

- OSPF
- BGP
- RIP (legacy)

Dynamic routing allows routers to exchange routing information automatically.

Although MikroTik supports these protocols, they should only be deployed when operational requirements justify the additional complexity.

---

# Site-to-Site Routing

Permanent VPN tunnels require routes on both ends.

Example:

```text
Headquarters
192.168.110.0/24

↓

WireGuard Tunnel

↓

Branch Office
192.168.210.0/24
```

Each router must know how to reach the remote subnet.

Missing routes are one of the most common causes of site-to-site VPN failures.

---

# Remote Access Routing

Remote users generally require access to:

- File Servers
- NAS Systems
- Microsoft 365 Hybrid Services
- SQL Servers
- Remote Desktop Servers
- Internal Web Applications

Only the required internal networks should be advertised through the VPN.

Avoid unnecessarily exposing entire internal address spaces.

---

# Policy-Based Routing (PBR)

Traditional routing decisions are based solely on the destination IP address.

Policy-Based Routing (PBR) extends this behavior by allowing routing decisions based on additional criteria such as:

- Source IP address
- Destination IP address
- Incoming interface
- Routing marks
- Firewall marks

Policy-Based Routing is particularly useful when multiple WAN connections or multiple VPN technologies coexist.

Typical use cases include:

- Routing administrative traffic through a dedicated VPN
- Directing backup traffic through a secondary Internet connection
- Separating guest and corporate traffic
- Multi-WAN environments

---

# Route Selection

When multiple routes exist for the same destination, RouterOS selects the preferred route according to several attributes.

The most important factors include:

- Administrative Distance
- Route Scope
- Route Status
- Gateway Availability

Understanding route selection is essential when implementing failover or redundant VPN gateways.

---

# Administrative Distance

Administrative Distance determines which route is preferred when multiple routes exist.

General recommendation:

| Route Type | Typical Distance |
|------------|-----------------:|
| Connected | 0 |
| Static | 1 |
| Dynamic | Protocol Dependent |

Lower values have higher priority.

Only modify route distances when the routing design specifically requires it.

---

# Multiple WAN Connections

Organizations frequently deploy multiple Internet providers for redundancy.

Typical topology:

```text
                ISP 1
                  │
                  ▼
             MikroTik
            /         \
           /           \
      ISP 2         VPN Users
```

Common deployment goals include:

- Automatic failover
- Load sharing
- Increased availability
- Reduced downtime

Routing policies should clearly define how VPN traffic behaves during WAN failures.

---

# VPN Failover

A production VPN infrastructure should continue operating even if the primary Internet connection fails.

Possible approaches include:

- Backup WAN
- Dynamic DNS
- Automatic gateway monitoring
- Recursive routing
- Netwatch automation

Failover should always be tested before production deployment.

---

# Equal-Cost Multi-Path (ECMP)

ECMP allows traffic to be distributed across multiple routes having the same routing cost.

Benefits include:

- Increased bandwidth
- Improved redundancy
- Better utilization of available links

However, ECMP should be evaluated carefully for VPN traffic because session persistence may be required.

---

# Hairpin NAT

Hairpin NAT becomes necessary when internal users access services through the organization's public IP address.

Typical scenario:

```text
Internal User

↓

Public DNS Name

↓

Public IP

↓

MikroTik

↓

Internal Server
```

Without proper Hairpin NAT configuration, internal access may fail even though external users connect successfully.

---

# Asymmetric Routing

Asymmetric routing occurs when packets travel through different paths in each direction.

Example:

```text
Client

↓

ISP 1

↓

MikroTik

↓

ISP 2

↓

Client
```

Possible symptoms include:

- Unstable VPN sessions
- Packet loss
- Connection resets
- Authentication failures

Routing should remain as symmetrical as possible.

---

# VLAN Routing

Organizations commonly separate networks using VLANs.

Example:

| VLAN | Purpose |
|------|---------|
| VLAN 10 | Users |
| VLAN 20 | Servers |
| VLAN 30 | Management |
| VLAN 40 | Guest |
| VLAN 50 | VPN |

VPN clients should receive access only to the VLANs required for their role.

Inter-VLAN communication should always be controlled by firewall policies.

---

# DNS Routing

DNS traffic deserves special attention.

VPN clients should resolve:

- Internal hostnames
- Active Directory records
- Internal services
- Application servers

using internal DNS servers whenever possible.

Avoid directing internal name resolution to public DNS providers.

---

# Route Summarization

As networks grow, route summarization helps reduce routing complexity.

Advantages include:

- Smaller routing tables
- Faster route lookups
- Easier troubleshooting
- Improved scalability

Summarization should be planned during network design rather than introduced later.

---

# Blackhole Routes

Blackhole routes intentionally discard traffic.

Typical use cases include:

- Preventing routing loops
- DDoS mitigation
- Bogon filtering
- Fail-safe routing

Use blackhole routes carefully and always document their purpose.

---

# Routing Security Best Practices

Routing plays a critical role in the overall security posture of a VPN infrastructure.

Incorrect routing may unintentionally expose internal networks, bypass firewall policies or create opportunities for lateral movement.

For production deployments always:

- Document every static route.
- Remove obsolete routes.
- Avoid overlapping IP networks.
- Restrict route advertisement to required networks only.
- Monitor routing table changes.
- Validate routing after every network modification.
- Review routing policies periodically.

Routing should always support the organization's security model rather than work around it.

---

# Enterprise Design Recommendations

A scalable routing design should prioritize simplicity, predictability and maintainability.

Recommended practices include:

- Dedicated VPN subnet
- Consistent IP addressing
- VLAN segmentation
- Static routing whenever practical
- Dynamic routing only when operationally justified
- Clearly documented routing policies
- Centralized monitoring
- Regular routing audits

As the infrastructure grows, a well-planned routing architecture significantly reduces operational complexity.

---

# Common Routing Mistakes

Many VPN connectivity issues originate from routing misconfigurations rather than VPN protocol failures.

Common mistakes include:

- Missing return routes
- Overlapping IP address ranges
- Incorrect default gateway
- Missing VPN subnet routes
- Split Tunnel configured incorrectly
- Full Tunnel configured unintentionally
- Incorrect Administrative Distance
- Missing firewall rules after adding routes
- Incorrect NAT configuration
- DNS servers located on unreachable networks

Most routing problems can be identified quickly by verifying each network path step by step.

---

# Troubleshooting Routing Issues

When diagnosing routing problems, verify the following in order:

1. VPN tunnel is established.
2. VPN client received the expected IP address.
3. Internal routes exist.
4. Return routes exist.
5. Firewall rules allow the traffic.
6. NAT rules are correct.
7. DNS resolution functions correctly.
8. Internal hosts respond to ICMP (where permitted).

Changing multiple routing components simultaneously should be avoided, as it complicates troubleshooting.

---

# Monitoring Recommendations

Routing should be continuously monitored in production environments.

Recommended monitoring items include:

- Route availability
- Gateway status
- WAN availability
- VPN tunnel status
- Interface utilization
- Packet loss
- Latency
- Routing table changes

Monitoring solutions such as Zabbix can detect routing anomalies before they impact users.

---

# Validation Checklist

Before deploying into production verify:

- IP addressing documented
- Static routes verified
- Dynamic routes validated (if used)
- VPN subnet reachable
- Return routes configured
- Firewall policies verified
- NAT reviewed
- DNS operational
- WAN failover tested
- Site-to-site connectivity verified
- Remote access tested
- Monitoring enabled
- Configuration backup completed

Every validation step should be completed successfully before production deployment.

# Summary

Reliable VPN connectivity depends on more than encryption and authentication.

A well-designed routing architecture ensures that VPN users can securely access only the resources they require while maintaining predictable traffic flows, minimizing operational complexity and supporting future growth.

Whether deploying WireGuard, OpenVPN or L2TP/IPsec, routing should always be designed as an integral part of the overall network architecture rather than as an afterthought.
